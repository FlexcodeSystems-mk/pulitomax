# Pulizia Pro

A premium, multilingual (Italian default, English, German) marketing website for "Pulizia Pro", a professional cleaning services company operating throughout Italy. Frontend-only React + Vite SPA.

## Run & Operate

- App runs via the `artifacts/cleaning: web` workflow (Vite dev server, binds to `PORT`)
- `pnpm --filter @workspace/cleaning run typecheck` — typecheck the cleaning artifact
- No backend / no database — all content is static and forms are client-only (no submission endpoint)

## Stack

- React 19 + Vite 7, TypeScript
- Routing: wouter (locale-prefixed, single flat router — no `nest`)
- i18n: custom React context (`src/lib/i18n.tsx`), locales it/en/de
- Forms: react-hook-form + zod (`@hookform/resolvers`)
- Animation: framer-motion
- Icons: lucide-react (brand/social icons also from lucide; `react-icons/si` for WhatsApp)
- Styling: Tailwind v4 (tokens via `hsl(var(--token))`), shadcn-style ui components

## Where things live

## SEO & Google Search Console

- `SITE.url` (in `src/lib/site.ts`) is the canonical production domain — UPDATE it to the real live domain before deploy.
- `public/sitemap.xml` + `public/robots.txt` are generated by `pnpm --filter @workspace/cleaning run sitemap` (script: `scripts/generate-sitemap.mjs`). Re-run after adding pages/services/blog posts (route lists are hardcoded in that script — keep them in sync).
- hreflang alternates (it/en/de + x-default) are injected per-page at runtime by the `Seo` component. Do NOT add static `<link rel="alternate" hreflang>` to `index.html` (causes duplicate/conflicting sets on non-home routes).
- Google Search Console verification: paste the `google-site-verification` meta tag into `index.html` (placeholder comment marks the spot), or verify via DNS / sitemap submission.

## Architecture decisions

- Locale routing uses a SINGLE flat wouter router (base = `BASE_URL`), NOT `nest`. `Router()` derives the locale from the first path segment; `LocalizedRouter` declares full locale-prefixed routes via `localePath(locale, ...)`. Do NOT reintroduce `nest`: links use absolute `localePath` hrefs (`/it/services`), and `nest` would prepend the locale base again → doubled prefixes (`/it/it/services`) and 404s on click / language switch.
- The `I18nProvider` is keyed by locale (`key={locale}`) so it remounts on language change and the locale is fixed by the URL, not stored state.
- Language switcher (`Navbar`): swap only the leading locale segment of wouter's base-stripped `useLocation()` location: `location.replace(/^\/[a-z]{2}(?=\/|$)/, "")`, then `navigate(\`/${next}${rest}\`)`.
- Forms have no backend: on submit they show a polished success state only.
- Per-page SEO is handled by the `Seo` component (title/meta/OG + JSON-LD injected into `<head>`).

## Product

Marketing site: home, 8 SEO-optimized service pages, subscription plans (Basic/Standard/Premium, placeholder prices), about, reviews, blog (5 categories, 10 full articles with FAQs), contact + quote forms, 20+ FAQs, WhatsApp + call floating buttons, newsletter signup, exit-intent popup, GDPR cookie consent. ## User preferences

- No emojis anywhere in the UI — use lucide-react icons.

## Gotchas

- Locale routing is a single flat wouter router — never reintroduce `nest` (links use absolute `localePath` hrefs, so `nest` doubles the locale prefix → `/it/it/...` and 404s on click).
- `react-icons/si` does not export `SiLinkedin` — use lucide `Linkedin` for social icons.
- Verify with `pnpm --filter @workspace/cleaning run typecheck`, not `build` (build needs workflow-provided env).
